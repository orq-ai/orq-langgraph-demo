# Plan: Refactor the Hybrid Data Agent demo from Toyota → Food Delivery

## Context

This repo is currently a LangGraph-on-orq.ai reference implementation set in the Toyota/Lexus vehicle sales + owner-manual domain. You're repositioning it as a **generic food-delivery onboarding demo** that can be shown to multiple food-delivery prospects without carrying any customer's brand.

The goal is to **keep 100% of the technical story intact** (LangGraph nodes, content_and_artifact tools, TypedDict state, OTEL tracing, evaluatorq pipeline, A/B prompts, managed-agent Approach B, bootstrap idempotency, `make doctor`) and **replace the domain layer only** — data, prompts, tools, eval cases, starters, and the copy in the docs.

**Intended outcome:** a feature-identical repo whose agent answers questions about *delivery orders, menus, dishes, restaurants, drivers, SLAs, refund policies, and food safety* instead of *vehicle sales and owner manuals*. Deployable via the same `make setup-workspace → make ingest-data → make run` flow.

**Confirmed decisions (from pre-plan Q&A):**

1. **Brand-neutral** — no single customer's brand baked in. The example data uses generic cuisine/city/dish names (Margherita, Berlin, Italian) which are real-world food terms, not brands. The agent's technical name stays `Hybrid Data Agent` — brand-neutral and already describes the pattern. No `DeliverEase`-style placeholder either; just "the delivery service" in prompts.
2. **Mixed-internal audience** — the same hybrid-agent story the current Toyota demo tells: new joiners, ops analysts, and support agents asking a mix of SQL-heavy and document-heavy questions. The 5/5/5 eval-dataset split (sql_only / document_only / mixed) stays as-is. This matches the "hybrid" positioning of the repo.

---

## Domain mapping (car → food)

The current structure is a **hybrid SQL + documents** agent. The food domain has an exact natural analog:

| Layer | Today | Food-delivery analog |
|---|---|---|
| Star schema fact table | `fact_sales` (vehicle sales) | `fact_orders` (delivery orders) |
| Dimensions | model, country, region, order type | dish, restaurant, city, cuisine |
| Primary entity | Vehicle (model + powertrain) | Dish (name + cuisine + allergens) |
| Grouping | Brand (Toyota, Lexus) | Restaurant partner |
| Unstructured docs | Owner's manuals, dealer contracts, warranty policy | Menu books, operations handbook, refund/SLA policy, food-safety policy, partner contract, customer-service playbook |
| Tool loop | "Show me sales + cite the manual" | "Show me orders + cite the policy/menu" |
| Router types | `toyota` / `more-info` / `general` | `on_topic` / `more-info` / `general` |

The *"hybrid data"* story — structured sales + unstructured policy, stitched by an agentic tool loop — transfers 1:1.

## New dataset design

### Structured (SQLite)

Star schema that mirrors the existing one so `sql_schemas.py` + `simple_sql_executor.py` stay structurally identical, only column names/types change:

- **`fact_orders`** — `order_id`, `dish_id`, `restaurant_id`, `city_id`, `driver_id`, `order_datetime`, `delivered_datetime`, `price_eur`, `rating`, `status`
- **`dim_dish`** — `dish_id`, `name`, `cuisine`, `category` (starter / main / dessert / drink), `base_price_eur`, `calories`, `allergens` (comma-separated)
- **`dim_restaurant`** — `restaurant_id`, `name`, `city_id`, `cuisine_type`, `avg_rating`
- **`dim_city`** — `city_id`, `name`, `country`, `region`

Populated from a synthetic CSV (generated by a new `scripts/generate_demo_orders.py` with a fixed random seed, ~5,000 orders across ~30 dishes × ~15 restaurants × ~10 cities × 18 months). The existing `scripts/structured_data_ingestion_pipeline.py` only needs its CSV path + column mapping swapped.

### Unstructured (orq.ai Knowledge Base)

6 PDFs totaling ~80 pages, all generated from Markdown sources committed under `docs/sources/*.md`. The **Markdown files and the rendered PDFs are both committed** to the repo — `scripts/generate_demo_pdfs.py` is provided to *regenerate* PDFs if you edit a source, not as a required install step.

Each doc uses one of two sourcing strategies:

- **Web-anchored** — paraphrase public-domain regulatory content into a shorter, delivery-company-flavored policy. Keeps terminology grounded in real language without any licensing issues.
- **Synthetic** — author the Markdown from scratch so we can deterministically seed answerable questions in the eval dataset.

| PDF | Sourcing strategy | Notes |
|---|---|---|
| `menu_book.pdf` | **Synthetic** | We author ~30 dishes with names, descriptions, ingredients, allergens, calories. Eval questions like *"what allergens does the Margherita contain?"* must be grep-findable in the Markdown. |
| `food_safety_and_hygiene_policy.pdf` | **Web-anchored** | Paraphrase the FDA Food Code / EU Regulation 852/2004 into a ~6-page internal policy. Covers temperature control, cross-contamination, packaging, PPE. |
| `allergen_labeling_policy.pdf` | **Web-anchored** | Paraphrase EU Regulation 1169/2011 (the top 14 allergens list + labeling rules). ~4 pages. |
| `delivery_operations_handbook.pdf` | **Synthetic** | Driver protocols, SLA tiers, pickup/dropoff flow, contactless delivery, dispute handling. |
| `refund_and_sla_policy.pdf` | **Synthetic** | Refund eligibility rules, late-delivery compensation tiers, missing-item handling. |
| `customer_service_playbook.pdf` | **Synthetic** | Escalation flow, complaint templates, refund decision tree. |

Rendering pipeline: `docs/sources/*.md` → `reportlab` (pure-Python, no system install needed) → `docs/*.pdf`. If `pandoc` is available on the machine, fall back to it for prettier output, but don't hard-require it.

## Phased execution — 3 phases, each internally shippable

### Phase 1 — Data + tools (~3-4 hours)

Goal: the agent runs end-to-end against the new SQL schema and produces correct answers, without touching docs or PDFs yet.

1. Write `scripts/generate_demo_orders.py` (deterministic synthetic CSV)
2. Rewrite `src/assistant/sql_schemas.py` (new schema + query types)
3. Rewrite `src/assistant/sql_tools.py` (9 renamed + retyped tools)
4. Update `tests/test_sql_tools.py` with new tool names + sample data
5. Run `make ingest-sql` → verify SQLite is populated
6. Update `src/assistant/prompts.py` `SYSTEM_PROMPT` to describe food tools
7. Minimal smoke test: `make run`, ask *"Top 5 dishes in Berlin"*, verify SQL tool dispatches correctly

Ship as commit `refactor(domain): swap SQL schema + tools from vehicle sales to delivery orders`.

### Phase 2 — Unstructured docs + prompts (~3-4 hours)

1. Write `docs/sources/*.md` (6 Markdown files, ~80 pages total). For the 2 web-anchored docs (food safety, allergen labeling), start by researching the cited public regulations (FDA Food Code, EU 852/2004, EU 1169/2011) and paraphrase into our voice. For the 4 synthetic docs, write from scratch with deterministic language so eval answers are grep-findable.
2. Write `scripts/generate_demo_pdfs.py` (render Markdown → PDF via `reportlab`; fall back to `pandoc` if available for prettier output). Commit both the `docs/sources/*.md` and the rendered `docs/*.pdf` so a clean clone doesn't need the script to run.
3. Run `make ingest-kb` against the new KB (create new KB in orq.ai to avoid polluting the old one)
4. Rewrite remaining prompts (`ROUTER_SYSTEM_PROMPT`, `MORE_INFO_SYSTEM_PROMPT`, `GENERAL_SYSTEM_PROMPT`)
5. Rename router classification `"toyota"` → `"on_topic"` throughout `graph.py`
6. Rewrite `scripts/setup_orq_workspace.py` inline prompts (system, variant-B, evaluators, managed agent instructions)
7. Rewrite eval dataset (`evals/datasets/tool_calling_evals.jsonl` — 15 rows)
8. Update `resources/conversation_starters.csv` (3 rows)
9. Run `make setup-workspace` against a NEW project on orq.ai (e.g. `delivery-demo`) — verify idempotent bootstrap
10. Run `make evals-run` — verify ≥90% tool-accuracy (may need one scorer-prompt pass)

Ship as commit `refactor(domain): swap PDFs, prompts, eval dataset for food-delivery demo`.

### Phase 3 — Documentation + screenshots (~2 hours)

## Data realism

All structured data and PDFs are **fully synthetic but use real-world food/geography names** (Margherita, Carbonara, Berlin, Milan, Amsterdam, Italian/Japanese cuisines). No public datasets — everything is deterministic and self-contained so the demo spins up in <10 minutes with no external data dependencies or licensing concerns. The synthetic generator uses a fixed random seed so a clean clone produces identical data every time.

Nothing in the generated content mentions a specific delivery company by name — prompts say "the delivery service" or "our operations" rather than a branded identity.
